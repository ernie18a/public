# m2 方法方向方案

## 核心判準

方法不是因為「看起來有效」就該做，而是要符合目前需求：

- 資料可集中取得
- 取得與處理成本低
- 可 API / CSV / 批次化
- CPU / RAM / RTX 4050 6G 可穩定執行
- 不依賴 HFT
- 不依賴企業內部深度資訊
- 可回測、可驗證、可替換
- 先用低成本資料建立 baseline，再決定是否擴資料

補充判斷：`OpenBB / EODHD / Finnhub / Twelve Data / FRED` 這類名稱只當「候選個案」，不是需求本身。真正需求是低成本、集中、自動化、可量化、低維護。

## 符合需求的主線

### 1. Cross-sectional / Tree-based / Feature Learning

定位：主模型。

核心方法：

- Cross-sectional ranking
- XGBoost / LightGBM / Random Forest
- tabular feature learning
- factor ranking
- multi-asset comparison

符合原因：

- 適合多商品比較，不只押單一方向
- 對 GPU 要求低，RTX 4050 6G 可承受
- 可用結構化資料批次處理
- 易做回測與 ablation
- 可接 portfolio / hedge / risk control

結論：

這是最符合目前硬體、人力、資料條件的主線。

### 2. Event Candidate Generation / 市場異常初篩

定位：第一層 cheap filter。

核心訊號：

- relative volume
- gap
- return dispersion
- volatility spike
- unusual options activity
- news spike

符合原因：

- 計算成本低
- 可先縮小研究 universe
- 避免對所有標的做昂貴深度分析
- 很適合接 cross-sectional model

結論：

應該早做，因為它直接降低後續所有方法的成本。

### 3. Vectorized Backtesting / Strategy Scanning

定位：驗證基礎設施。

核心工具：

- vectorbt
- pandas / NumPy / Numba
- rule-based strategy scanner

符合原因：

- 解決人力不足下的實驗效率
- 不需要大 GPU
- 可快速淘汰無效訊號
- 讓方法選擇基於回測結果

結論：

不是 alpha 方法，但屬於高邊際效益必要基礎設施。

### 4. Regime Detection

定位：輔助核心，不當主 alpha。

用途：

- 判斷市場狀態
- 決定模型是否啟用
- 調整倉位與風險
- 對商品或市場做分群

符合原因：

- 可用價格、波動、相關性、宏觀資料完成
- 不要求大 GPU
- 作為 gating system 邊際效益高

結論：

適合作為主線模型的 gating layer。

### 5. Change Point Detection

定位：波段切分工具。

核心方法：

- PELT
- Window-based change point
- Kernel change point
- Bayesian online change point detection

符合原因：

- 計算成本可控
- 不依賴深度學習
- 能把任意窗口轉成較有意義的波段
- 可接 XGBoost / Matrix Profile / Event Study

結論：

是低成本提高資料結構品質的方法，應放在早期。

### 6. Matrix Profile / Shape Pattern Mining

定位：相似波段與重複形狀搜尋。

核心方法：

- Matrix Profile
- STUMPY
- motif discovery
- discord detection

符合原因：

- 比手工 K 線型態更可驗證
- 可先小規模實驗
- 可和 Change Point 結合
- 維護成本低於大量 alternative data 爬蟲

限制：

- 多商品全量掃描可能吃記憶體
- 需要控制窗口與候選集合

結論：

適合作為 pattern discovery 工具，不單獨當策略。

### 7. Portfolio / Risk Layer

定位：把訊號轉成可交易組合。

核心控制：

- exposure control
- sector neutrality
- factor neutrality
- volatility targeting
- position sizing
- correlation control
- turnover control

符合原因：

- 計算成本低
- 對個人邊際效益高
- 不需要硬體優勢
- 可避免好訊號被倉位與暴露破壞

結論：

應作為主線必要配套，不是最後才補。

## 候補 / 延後加入

### 8. DTW / Time Series Clustering

符合部分：

- 可找相似走勢
- 可做波段或商品分群
- 可解釋

延後原因：

- 大量 pairwise comparison 成本高
- 多商品、多窗口時容易變慢
- 應排在 Change Point / Matrix Profile 之後

### 9. Probabilistic Forecast

符合部分：

- 能處理報酬分布與不確定性
- 可協助 position sizing
- 可與 tree model 結合

延後原因：

- 驗證比方向分類複雜
- 需要校準分布品質
- 初期不應壓過主線 baseline

### 10. tsfresh / sktime 類工具

符合部分：

- 可自動抽時間序列特徵
- 可降低人工特徵設計成本
- 可服務 tree-based model

延後原因：

- 可能產生大量無效特徵
- 需要嚴格 feature selection
- 框架複雜度不能超過收益

### 11. Event Study / 事件前兆反推

符合部分：

- 把大波段反推成可驗證前兆
- CPU 批次即可完成
- 可接 Change Point / XGBoost

延後原因：

- 事件定義會強烈影響結果
- 容易 data snooping
- 必須先建立基礎回測與樣本外驗證

### 12. LLM Semantic Feature

定位：把非結構化資訊轉成 feature，不讓 LLM 直接預測價格。

可接受用途：

- catalyst detection
- risk factor change
- supply chain fragility
- transcript / filing 摘要成結構化欄位

延後原因：

- 成本容易爆炸
- 需要 hallucination check
- 只能對候選清單做，不做全市場爆搜

### 13. Small Representation Learning

可接受用途：

- autoencoder
- 小型 CNN
- sequence embedding
- 把波段壓成 embedding 後餵給 tree model

延後原因：

- 必須和 tree baseline 比較
- 沒有明顯提升就停止
- 不做大型 end-to-end 預測

### 14. Conservative Causal Discovery

可接受用途：

- lead-lag 假說生成
- feature pruning
- 檢查 macro / sector / liquidity 干擾

延後原因：

- 金融資料非平穩
- 不能把結果直接當交易規則
- 早期只適合作研究輔助

### 15. TabPFN / 小樣本 Tabular Foundation Model

可接受用途：

- 小樣本 tabular 對照組
- 與 XGBoost baseline 比較

延後原因：

- 不適合當主線
- 對資料規模與分布限制較多
- 實務穩定性需要驗證

## 不符合需求 / 不符合原因

### 1. End-to-end Sequence Model

代表：

- LSTM
- Transformer sequence predictor
- large temporal model

不符合原因：

- 資料量要求高
- 調參成本高
- 容易 overfit
- RTX 4050 6G 容易受 batch size、sequence length、模型大小限制
- 輸出不一定比 tree-based baseline 穩

結論：

不做「Transformer 預測明天漲跌」這種主線。只允許小型 embedding 實驗。

### 2. Reinforcement Learning

不符合原因：

- action space 大
- reward design 困難
- portfolio constraint 複雜
- regime 變化會讓訓練不穩
- 很容易變研究黑洞
- 對人力、驗證、模擬環境要求高

結論：

目前不當主線，也不當早期實驗。

### 3. HFT / Ultra-low Latency 方法

不符合原因：

- 硬體不匹配
- 網路與 colocated infrastructure 不匹配
- 資料延遲與交易執行條件不匹配
- 個人研究環境無法形成優勢

結論：

直接排除。

### 4. 企業內部深度資訊方法

代表：

- 公司內部訂單
- 未公開供應鏈細節
- procurement leaks
- 深度專家網絡

不符合原因：

- 資訊不可穩定取得
- 驗證困難
- 合規與可靠性風險高
- 無法形成可重複 pipeline

結論：

不納入方法主線。

### 5. 高維護 Alternative Data 深挖

代表：

- 分散社群爬蟲
- 原始海關資料深挖
- 大量拆機報告整合
- LinkedIn hiring 大量追蹤
- 多來源 shipping / customs parsing

不符合原因：

- 資料不集中
- 需要維護多個爬蟲
- entity matching 成本高
- 清洗成本高
- 失敗點多
- 邊際效益不確定

結論：

目前只保留已結構化、集中、低維護的來源；分散深挖先排除。

### 6. Raw Transcript / IR / ESG 全市場抓取

不符合原因：

- 每家公司來源不同
- 下載與格式維護成本高
- PDF、影音、網頁混雜
- 需要大量 LLM 或人工處理

結論：

只有「集中聚合版」符合需求；逐家公司抓取不符合。

### 7. 社群情緒全量分析

代表：

- Reddit
- PTT
- Dcard
- 分散論壇資料

不符合原因：

- 雜訊高
- 清洗成本高
- 合規與 rate limit 風險
- 與供應鏈/BOM 主題距離遠

結論：

不當早期主線；最多作為特定事件的人工驗證材料。

## 建議落地順序

### Phase 1: 建立可驗證基準

- 集中市場資料與官方資料
- Event candidate filter
- 基礎特徵
- Vectorized backtesting
- Cross-sectional tree-based baseline
- Portfolio risk layer

### Phase 2: 加入市場狀態與波段結構

- Regime Detection
- Change Point Detection
- 波段特徵
- 不同 regime 下的模型穩定性測試

### Phase 3: 加入相似形狀與分布預測

- Matrix Profile
- DTW / clustering 小範圍實驗
- Probabilistic Forecast
- Event Study 反推前兆

### Phase 4: 小規模新方法對照

- TabPFN
- Small representation learning
- LLM semantic feature
- Conservative causal discovery

## 最小可行主架構

```text
集中資料源
-> Event candidate filter
-> 基礎特徵 / tsfresh 選用
-> Regime gating
-> Change Point 波段切分
-> Cross-sectional Tree-based model
-> Vectorized backtest
-> Portfolio risk control
-> LLM semantic feature 選用
```

## 一句話結論

目前最符合需求的是：

```text
Cross-sectional Tree-based baseline
+ Event candidate filter
+ Regime gating
+ Change Point / Matrix Profile
+ Vectorized backtesting
+ Portfolio risk layer
```

目前應排除的是：

```text
HFT
RL
大型 end-to-end sequence model
企業內部深度資訊方法
高維護分散 alternative data 深挖
Raw transcript / IR / ESG 全市場抓取
社群情緒全量分析
```
