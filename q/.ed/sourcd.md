# data 來源

這份資料清單可以壓成三種邏輯：

## 1. API 或現成 CSV，直接取
實際在做什麼：

資料本身已經整理好，重點是穩定拿到。  
怎麼用：

直接用官方 API 或 CSV 下載。  
適合放進基礎資料層。  
本質：低維護成本，適合當主資料源。

包含：

- Stooq
- Alpaca
- Alpha Vantage
- FRED
- FMP
- Wikimedia
- Hacker News
- GitHub
- Google Trends
- USAspending.gov
- OpenSecrets
- Polygon

## 2. 集中單站，需爬蟲
實際在做什麼：

資料集中在少數網站，但沒有乾淨 API 或歷史接口不完整。  
怎麼用：

用爬蟲定期抓取，必要時補上 API。  
適合放進特定主題資料層。  
本質：可取，但要承擔網站結構變動。

包含：

- CapitolTrades / QuiverQuant
- OpenInsider
- iShares / SPDR
- earningscall.biz
- App Store / Google Play
- Reddit

## 3. 分散各處，難維護
實際在做什麼：

資料分布很散，格式差異大，沒有單一穩定入口。  
怎麼用：

只能做專案型抓取，不適合當長期核心管線。  
本質：成本高，維護風險大。

包含：

- 企業招聘資訊
- SEC EDGAR 原始 Filing
- 專利資料
- 企業 IR 官網
- 航運與供應鏈資料

## 核心統一理解

這份資料清單其實可以壓成三種邏輯：

- 直接取 -> API / CSV
- 集中抓 -> 單站爬蟲
- 分散整合 -> 高維護成本資料

## 最實際可落地的一條路

- 先把主資料源固定在 API / CSV
- 再補少量高價值爬蟲站
- 最後才碰分散資料

## 後半段

不是先追求資料量，而是先決定資料要放在哪一層。

這一段本質上是：

- 資料分層
- 可維護性
- 取得成本控制

## 可行方法

1. 基礎價格資料
2. 宏觀資料
3. 公司基本面
4. 社群 / 注意力資料
5. 事件資料
6. 分散式補充資料

## 研究流程

- 先定義主問題
- 找最穩定的資料源
- 把資料切成主層與輔層
- 只補對研究有明確增益的來源
- 避免為了資料而資料

## 主要風險

- 網站結構變動
- API rate limit
- 資料缺口
- 維護成本失控

## 真正重點

不是把所有資料都抓下來，而是：

- 先確定哪類資料真的會進模型
- 再決定值不值得維護
