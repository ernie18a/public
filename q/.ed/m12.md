# 方法整合

## 核心判準
- 邊際效益 = 預期產出 - 取得成本 - 計算成本 - 維護成本 - 資訊成本。
- 只要高度依賴企業內部資訊、深度內餡、或高頻延遲優勢，就直接排除。
- 只要需求超出 `CPU / 記憶體 / RTX 4050 6G` 的穩定可承受範圍，就直接排除。
- 目標是可重複、可維護、可擴充，不追名詞新。

## Tier 1: 主線

### 1. Change Point / Regime Detection
- 目的：先把連續序列切成有意義的波段或狀態。
- 作用：降低噪音，讓後續模型只看同一種市場環境。
- 適合：價格序列、波動切換、趨勢轉折、風險切換。
- 判定：這是最值得先做的入口。

### 2. Cross-sectional / Tree-based
- 目的：用多商品、多特徵做排序、分類、回歸。
- 作用：適合中頻、非單一方向、需要組合多個訊號的情境。
- 優點：硬體要求低，XGBoost / Random Forest / LightGBM 類型都可控。
- 判定：這應該是主模型方向，不要先跳深度模型。

### 3. Matrix Profile / DTW / Pattern Search
- 目的：找歷史中相似波段與重複型態。
- 作用：做前兆探索、型態比對、相似區段搜尋。
- 限制：資料太分散或太髒時，邊際效益會快速下降。
- 判定：適合當輔助主線，不適合單獨當核心。

### 4. Probabilistic / Bayesian
- 目的：處理不確定性、狀態切換、分布預測。
- 作用：比單點預測更符合風險控制需求。
- 優點：通常比重型深度模型輕，適合目前硬體。
- 判定：可以放在主線旁邊，不要做成重工程系統。

## Tier 2: 可做但要收斂

### 5. Feature Learning
- 目的：把序列壓成可用特徵。
- 可接受：小型 autoencoder、簡單 embedding、`tsfresh` 類特徵抽取。
- 不建議：大模型端到端直接吃原始序列。
- 判定：只能服務主線，不要獨立成黑盒。

### 6. Backtest / Simulation
- 目的：驗證方法是否真的有邊際效益。
- 作用：快速淘汰看起來漂亮但實際沒用的方法。
- 判定：不是附屬品，是方法選擇的一部分。

### 7. Tool-driven Research Stack
- `OpenBB`：資料整合入口。
- `tsfresh`：時間序列特徵抽取。
- `vectorbt`：向量化回測與策略掃描。
- `sktime`：時間序列 ML 框架。
- `backtrader`：交易流程與模擬。
- 判定：這些是研究基礎設施，不是 alpha 本身。

## Tier 3: 直接排除

### 8. RL
- 原因：狀態、動作、獎勵都太容易失真，維護成本高。

### 9. HFT
- 原因：硬體、延遲、交易基礎設施都不符合現狀。

### 10. 依賴企業內部資料的深內餡方法
- 原因：資訊取得成本超出可承受範圍。

### 11. 重型深度序列模型
- 原因：訓練、調參、驗證成本太高，6G 硬體不划算。

## 來源配置
- `Change Point / Regime Detection`：先吃 `Stooq / Polygon / Alpaca / Alpha Vantage / FRED / FMP`
- `Cross-sectional / Tree-based`：先吃 `Stooq / Polygon / Alpaca / Alpha Vantage / FRED / FMP`
- `Matrix Profile / DTW`：先吃上述結構化來源，再視需要加 `iShares / SPDR`
- `Probabilistic / Bayesian`：先吃上述結構化來源
- `Feature Learning`：先吃上述結構化來源，避免先擴到高維護來源
- `Sequence / RL`：先不要擴大資料源，因為主問題不在資料量而在方法成本

## 最佳主線
- `Change Point / Regime Detection`
- `Cross-sectional + Tree-based`
- `Matrix Profile` 做輔助搜尋
- `Probabilistic / Bayesian` 做風險與狀態補強

## 一句話
- 先切結構，再做橫截面，再用輕量模型驗證。
- 任何需要重硬體、重人力、重內部資訊的方法，直接 drop。
